Global RBAC Permission Matrix
Every permission in MedSync is enforced at three layers: JWT role field (signed RS256), server-side PERMISSION_MAP check, and role-based serializer on response. The table below is the single source of truth.

Role 02  Nurse
Scope: Ward-scoped only · ward_id in JWT · extracted server-side
Vitals · Dispensing · SBAR Handover · Shift tracking · Ward bed grid
Pages & Routes
Dashboard — Stat Cards
Ward Bed Grid
4-column grid of bed cards. Each bed shows: bed code · patient name · age/gender · admission date · last vitals timestamp · status colour.
Status — left border accent: green=stable · amber=watch · red=critical · gray=vacant
stable: all vitals within normal, no active alerts
watch: vitals overdue >4h OR any out-of-range (non-critical) vital
critical: active ClinicalAlert OR SpO2 <88% OR BP systolic >180
Bed card shows "Vitals overdue Xh" in red when last_vitals_at > 4h ago
1-click actions per bed card: [Vitals] [Chart] [Note] [Dispense] (Dispense only shown if pending prescriptions exist)
Nurse NEVER sees diagnosis in free text. Patient view shows "Admitted for [department]" only — NOT ICD-10 codes.
Vitals Entry Form
Touch targets: minimum 44×44px — nurses enter on tablets at bedside
Auto-advance: Temp → Pulse → Resp → BP sys → BP dia → SpO2 → Weight
Critical vitals modal: if SpO2 <88% or other critical → full-screen modal. Cannot dismiss without confirming action taken. Logs CRITICAL_VITALS_CONFIRMED audit event.
Batch mode: toggle to record vitals for all ward patients in one table. POST /records/vitals/batch — server rejects any patient outside ward with 403.
Dispensing
Prescription card shows: patient name · bed · drug name · dosage · frequency · route · written by · time since prescribed
If allergy_conflict=true → red "⚠ ALLERGY CONFLICT" badge. Nurse can still dispense (override already authorised by doctor). Tooltip shows override reason.
[Mark Dispensed] → POST /records/prescription/:id/dispense. Records dispensed_by + dispensed_at.
Filters: Pending / Dispensed today / All. Sorted oldest first.
SBAR Handover
Shift Tracking Widget (Sidebar)
Persistent in sidebar across all nurse pages — NOT a dashboard panel
Shows: shift name · ward · current time · time remaining · progress bar
[Log break] → POST /shifts/:id/break → pauses accountability. Button changes to [Resume]
[End shift] → navigates to /worklist/handover — always triggers SBAR handover, never silent logout

Role 03  Lab Technician
Scope: Lab-unit scoped · lab_unit_id in JWT · extracted server-side
Order queue · Result entry · TAT tracking · PDF upload · No clinical data visible
Pages & Routes
Dashboard — Stat Cards
Order Queue — Sort Rules
STAT orders are ALWAYS at the top regardless of creation time. Even if STAT arrives after 20 routine orders — it surfaces immediately to position 1. Sort: urgency_rank ASC (stat=1, urgent=2, routine=3), then created_at ASC.
Order row columns: urgency badge · test name · patient name + GHA ID (NO diagnoses) · TAT countdown · status badge · ordering doctor · [Open →] button
TAT countdown: "Due in 42 min" — turns red when <30 min remain. Shows "OVERDUE Xm" in red when past threshold.
TAT targets: STAT=60 min · Urgent=4h · Routine=24h
Filter tabs: All / Pending / In Progress / Resulted (today) / Verified
Status badge flow: Ordered → Collected → In Progress → Resulted → Verified
Result Entry — 6-Step Flow
TAT Metrics Dashboard
Average TAT today: per urgency type with progress bar vs target. Turns red if avg > target.
TAT breaches today: count + list of specific tests that breached with overdue time
Pending by age: bar chart — 0-1h (green) · 1-4h (amber) · 4-24h (amber) · 24h+ (red)
Throughput: Today / Yesterday / 7-day average as stat cards
Data Visibility Rules
Lab Technician NEVER sees clinical data. The lab order detail page shows: patient name, age, gender, GHA ID, test name, urgency, ordering doctor. It NEVER shows diagnoses, prescriptions, nursing notes, encounter notes, or allergy details. Enforced by LabOrderSerializer on backend.
Lab Technician API Endpoints
Lab Technician Cannot Do
See diagnoses, prescriptions, nursing notes, encounter notes, or allergy details
Modify or cancel lab orders (can only update status and enter results)
Verify results (verification is doctor action — PATCH /lab/orders/:id to "verified")
Access patients outside their assigned lab unit
Run AI analysis
Register or search patients
Access admin pages or audit logs
Role 04  Receptionist
Scope: Hospital-scoped · Demographics ONLY · Zero clinical data access
Appointment scheduling · Patient check-in · Demographics search · No clinical access
Pages & Routes
Dashboard — Stat Cards
Appointment Management
Book: Search patient → Select department → Select doctor (optional) → Date/time → Notes → Save
Double-booking prevention: as soon as date + time + department selected, check availability BEFORE save. Show "Dr. X already has a patient at HH:MM. Available: HH:MM · HH:MM" with clickable slot chips.
Reschedule: click appointment → new date picker → confirm. PATCH /appointments/:id { appointment_date }
Cancel: click → reason input (required, blocks confirm if empty) → confirm. DELETE /appointments/:id { cancellation_reason }
Check-in: ONE click from appointment row. Toast only — NO confirmation modal. Validates ±2 hours of scheduled time.
No-show: manual button appears after 30 min past scheduled time. Auto-mark via Celery task every 15 min.
Calendar colour coding — consistent across List/Day/Week: scheduled=gray · checked_in=green · completed=blue · no_show=red · cancelled=strikethrough
Check-In Rule
SPEC RULE: Check-in must be achievable in 2 clicks from dashboard. Click 1: "Check in" button. Click 2: none — action fires immediately. Toast confirmation only. No modal. This is non-negotiable for front-desk efficiency.
Patient Search
Search by: Full name (partial match) / Ghana Health ID / Phone number / NHIS number
Results in under 300ms. Debounce at 150ms.
Patient card shows: Full name · Age · Gender · GHA ID · NHIS · Phone · Registered date
Buttons: [Book Appointment] and [View Demographics] — no clinical links
Backend enforces DemographicsOnlySerializer for receptionist role. Even direct API call to /patients/:id/records returns 403. No diagnoses, prescriptions, vitals, labs, or allergies ever returned.
If patient not found: "Patient not found. To register a new patient, please ask a doctor or hospital admin." NO register form shown — receptionist cannot register patients.
Receptionist API Endpoints
Receptionist Cannot Do
View any clinical data (diagnoses, prescriptions, vitals, labs, allergies, encounters)
Register new patients
Modify patient demographics
Access /patients/:id/records (returns 403)
View audit logs
Create or manage staff accounts
Access admin or clinical pages

AI Integration Module
AI decision support is available ONLY to the Doctor role. All other roles have zero access to AI analysis features. AI is a decision support tool — it never takes autonomous action.
AI Analysis Types
AI UX Rules — Non-Negotiable
AI suggestions NEVER auto-fill form fields. They appear in a right-side panel with individual "Apply to form" buttons. The doctor must take explicit action for every suggestion. AI does not write to fields autonomously under any circumstance.
"Run AI Analysis" button in encounter form header → POST /ai/analysis { patient_id, analysis_type: "comprehensive" }
Results panel slides in from RIGHT — does not replace the SOAP form
Panel shows: risk scores · diagnosis suggestions ranked 1-10 · triage level 1-5 · similar past cases · referral recommendations
Each diagnosis suggestion has [Apply to form] button → pre-fills ICD-10 search field only (doctor must still confirm)
AI History tab on patient record: all past analyses with timestamp, type, confidence score, expandable results
Average response time target: <800ms. If >800ms → AI service shows amber warning on Super Admin dashboard.
AI Endpoints
AI Does Not Have Access To
The AI module does not have direct database access
AI cannot write to any patient record
AI suggestions are advisory only — no clinical decision is made without doctor confirmation
AI analysis results are not shared between hospitals — each analysis is scoped to the requesting hospital
Security Architecture
Authentication Stack
Account Security Rules
Account lockout: 5 failed attempts → locked 15 minutes. Admin notified. ACCOUNT_LOCKOUT audit event.
New device login: email + SMS alert even if MFA succeeded. "New login detected from [location] at [time]."
Screen lock: 5 minutes inactivity on clinical workstations → blur page + PIN re-entry. Session not destroyed.
MFA enrolment: mandatory on first login. Blocking setup screen — no skip option. Admin cannot grant PHI access until MFA complete.
Backup codes: single-use. BACKUP_CODE_USED audit event. Alert when <3 remain. Regeneration requires TOTP re-verification.
Password Recovery Tiers
Consent Management
Every inter-hospital data request requires explicit, scoped, time-limited patient consent. Patients control access via OTP-verified SMS. No clinical need bypasses this — except documented break-glass.
Consent Lifecycle
Consent Rules
Consent is SCOPED — not blanket. Doctor must specify: which hospital, which record types (encounters/labs/prescriptions/all), for how long.
Patient is the consent authority — not the doctor, not the admin.
Consent is revocable at any time. Revocation creates CONSENT_REVOKED audit at both hospitals.
Expiring consent shows countdown to doctor: "Consent expires in 2h 14m — [Extend]". Extend sends new OTP.
Consent page for patient requires no account — OTP link works in any mobile browser. Shows requesting doctor, hospital, exact record types, duration in plain language.

Audit Trail & Data Integrity
Chain Hash Design
content_hash = SHA256(action + user_id + patient_id + timestamp + details)
chain_hash = SHA256(content_hash + previous_record.chain_hash)
Any retroactive modification invalidates all subsequent hashes — tampering is mathematically detectable
AuditLog table: INSERT-only permission at database level. No UPDATE or DELETE ever permitted.
Chain integrity validation: POST /audit/validate-chain (Super Admin only) — re-computes hashes for date range. Returns tampered_records list.
Audit Event Catalogue
Global Patient Identity (GPID)
Architecture
Format: GHA-XXXXXXXXX (GHA prefix + 9-character alphanumeric). Immutable once assigned.
GlobalPatient record (central): gpid · full_name · dob · gender · national_id · nhis_number · registered_hospitals · created_at. No clinical PHI.
LocalPatient record (per hospital): hospital_patient_id · gpid (FK) · blood_group · allergies · hospital_specific_notes.
Clinical records (encounters, labs, prescriptions) link to LocalPatient — not GlobalPatient. Hospital data sovereignty preserved.
Duplicate Detection
At registration, system performs probabilistic match on: national_id (exact) · nhis_number (exact) · name + DOB (fuzzy, Levenshtein ≤2)
If potential match found: "A patient matching this profile exists in the network. Is this the same person?" Explicit confirmation required before GPID reuse.
GPID Merge (Super Admin Only)
POST /global-patients/merge { primary_gpid, secondary_gpid, reason, patient_otp }
Patient OTP required before merge — prevents accidental or malicious merging
Secondary GPID is deprecated (not deleted) — remains searchable and redirects to primary
Creates GPID_MERGE audit event

User Creation & Access Control Workflow
4-Phase Onboarding Flow
Invitation Token Rules
Token lifetime: 72 hours (configurable via INVITATION_TOKEN_HOURS in settings.py)
Celery task runs every 6 hours — notifies admin when tokens expiring within 12 hours
Expired tokens cannot activate — new token required via Resend
Bulk resend: POST /admin/users/bulk-resend-invites — regenerates all expired tokens for pending users
Bulk Import Rules
Maximum 500 users per CSV file
dry_run=true: validates all rows, returns preview + errors WITHOUT creating any users
Atomic: wrapped in transaction.atomic() — if any row fails, ALL rows are rolled back. No partial imports.
Error response: returns ALL errors before creating any users. Admin must fix and re-upload.
Inter-Hospital Record Access
Architecture — Central Platform as Mediator
Neither hospital directly queries the other's database. ALL cross-hospital requests flow through MedSync central platform.
Central platform: (1) verifies requesting user's RBAC role, (2) checks active consent token, (3) retrieves records from source hospital API, (4) applies semantic normalisation, (5) returns only consented scope.
Semantic normalisation: ICD-10 codes verified, drug names resolved to INN, lab units converted to SI.
Data sovereignty: receiving hospital NEVER stores cross-hospital data. Retrieved on-demand, displayed in session only.
Cross-Hospital Access Flow
Break-Glass Emergency Access
POST /break-glass/access { patient_gpid, source_hospital_id, reason (min 50 chars) }
Reason is free text — no dropdown. Forces genuine clinical justification.
Access token valid 4 hours only. Full record scope returned (not consent-scoped).
BREAK_GLASS_ACCESS audit at BOTH hospitals. Super Admin notified immediately.
Must be reviewed by Super Admin within 24 hours. Unreviewed events → compliance flag.
Abuse detection: >3 uses in 24h by same user → excessive_usage flag. Hospital Admin notified. Only Super Admin controls threshold.

Layer
Implementation
Identity
Unique user_id — never recycled. Tied to all audit events.
Authentication
Password: min 12 chars, upper+lower+digit+symbol. bcrypt cost factor 12. Last 5 hashes tracked — no reuse. MFA: TOTP mandatory for all PHI roles. 8 backup codes. No skip option.
Authorisation
JWT RS256: user_id · role · hospital_id · ward_id (nurses) · issued_at · expires_at (15 min). Refresh token (7 days) in HttpOnly cookie.
Session
Access token expires 15 min inactivity. Clinical workstations auto-lock after 5 min. Session token revocation list for immediate logout.
Inter-hospital
X-Consent-Token (JWT RS256) signed by central platform. Hospital nodes hold only public key — cannot forge tokens.
Tier
Flow
Tier 1 — Self-service
POST /auth/forgot-password { email } → reset link valid 1h. Always returns 200 (anti-enumeration). POST /auth/reset-password { token, new_password }. Validates last 5 hashes.
Tier 2 — Admin reset
POST /admin/users/:id/send-password-reset (Hospital Admin → their staff). Sends reset email. Logged as PASSWORD_RESET_INITIATED.
Tier 3 — Force reset
POST /superadmin/users/:id/force-password-reset (Super Admin → any user). Immediately invalidates password. Logged as FORCE_PASSWORD_RESET.
Step
Detail
1. Doctor initiates
POST /consent/request { patient_gpid, granting_hospital_id, scope, reason, expires_in_hours }. Status: pending_patient_approval.
2. OTP to patient
SMS: "Dr. [Name] at [Hospital B] is requesting access to your records from [Hospital A]. Enter OTP [XXXXXX] to approve. Expires in [X hours]."
3. Patient responds
PATCH /consent/request/:id/respond { otp, decision }. No login required — OTP is the auth. Status: active or denied.
4. Token issued
Signed X-Consent-Token (JWT RS256): patient GPID · granting hospital · receiving hospital · scope · expiry. Sent to doctor's session.
5. Scoped access
Each cross-hospital request carries X-Consent-Token. Receiving hospital verifies signature and scope before returning any records.
6. Expiry/revoke
Token auto-expires. Patient can revoke early via SMS portal. Revocation invalidates token in central revocation list within seconds.

Layer
Method & Detail
At rest — DB level
Transparent Database Encryption (TDE) — AES-256 encrypts entire DB at storage layer. Raw files unreadable without TDE master key.
At rest — field level
Application-layer AES-256 on: national_id · nhis_number · phone_number · allergy details · HIV/mental health flags. Per-record key. Defence-in-depth.
In transit — external
TLS 1.3. TLS 1.0 and 1.1 disabled. HSTS headers. Internal service-to-service uses mutual TLS (mTLS).
Consent tokens
RS256 JWT. Private key in central KMS only. Hospital nodes hold public key only — cannot forge tokens.
Backups
AES-256 with separate backup key. Stored off-site. Recovery test monthly. Failed recovery = critical incident.
Key management
Centralised KMS. App servers never have direct KMS access — request operations via authenticated API. Key rotation annually or on suspected compromise.

Phase
Steps
Phase 1 — Super Admin setup
POST /superadmin/onboard-hospital (name, region, NHIS code). POST /admin/users/invite { email, role: hospital_admin, hospital_id }. 72h token generated. TOTP secret created.
Phase 2 — Hospital Admin activation
GET /auth/activate-setup?token= → TOTP QR code. Scan with Google Authenticator or Authy. Set password (12+ chars). POST /auth/activate { token, password, totp_code }. Returns JWT. 8 backup codes issued.
Phase 3 — Staff creation
Single: POST /admin/users/invite { email, role, department, ward, lab_unit }. Bulk: POST /admin/users/bulk-import (CSV, max 500, atomic). 72h tokens. Auto-scoped to admin's hospital.
Phase 4 — Staff activation
Identical to Phase 2. Mandatory TOTP for all roles. No skip option. Daily login: email + password + 6-digit TOTP → JWT access (15 min) + refresh (7 days HttpOnly).
Step
System Action
1. Patient lookup
GET /cross-facility/patient-lookup?gpid= — returns masked name + hospital list. No clinical data.
2. Consent request
POST /consent/request — doctor selects hospital + scope. Patient receives OTP.
3. RBAC verification
Platform verifies: requesting user is doctor + account active + hospital in good standing.
4. Consent-gated query
GET /cross-facility/records with X-Consent-Token header. Platform routes to source hospital, returns normalised data.
5. Unified display
Cross-hospital records shown alongside local records. Labelled with source hospital. Read-only.
6. Dual audit
CROSS_HOSPITAL_RECORD_ACCESSED event at BOTH hospitals. Includes: requesting user, record type, consent token ID, timestamp.
